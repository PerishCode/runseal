# v0.4.6

- Add per-invocation log files under `ENVLOCK_LOG_HOME` or `ENVLOCK_HOME/logs`, with unified Rust CLI, plugin host, and `node.sh` traceability.
- Harden `envlock plugin node` across stale-lock recovery, read-only state dirs, unwritable log dirs, and failure-path diagnostics.
- Support GNU libc Linux on both `x86_64` and `aarch64`, and validate install-based release behavior on published artifacts.
- Improve multi-Node coexistence with mixed `npm`/`pnpm`/`yarn` versions, including global installs and package-manager-managed projects.
- Align manager routing with project-level `packageManager` and corepack behavior so selected Node routing and versioned cache/store/global paths stay coherent.

## Validation

- Basis:
  - published-artifact install validation against `v0.4.6-beta.10`
  - GNU libc Linux on `x86_64` and `aarch64`
  - local fully isolated rehearsal on macOS with temporary `HOME`, `ENVLOCK_HOME`, `ENVLOCK_LOG_HOME`, projects, state, and toolchains
- Multi-Node coexistence validation with `Node 18 / 22 / 24` installed side-by-side, hostile PATH ordering, and explicit selection of a different runtime Node.
- Validation across `npm`, `pnpm`, and `yarn` with mismatched manager versions, manager shims, and checks that manager-invoked Node still resolved to the selected runtime.
- Global install validation for `npm install -g`, `pnpm add -g`, `pnpm dlx`, and `yarn global add`, verifying binaries were installed under envlock-managed bin paths and remained runnable.
- Project-level `packageManager` validation with corepack-managed `pnpm` and `yarn`, confirming that apply succeeded without being confused by project-local manager metadata while runtime cache/store/global paths followed the project-selected manager version.
- Validation of stale locks, interrupted apply, read-only state dirs, unwritable log homes, and missing, non-executable, or looped manager binaries.
- Validation with spaces in `ENVLOCK_HOME`, `ENVLOCK_LOG_HOME`, `--state-dir`, and manager binary paths, confirming init/apply and manager invocations still succeeded.

### Concrete runs

- Published artifact install on GNU libc Linux:
  - `sh scripts/install.sh --version v0.4.6-beta.10`
  - `envlock plugin node init --state-dir ...`
  - `envlock plugin node apply --state-dir ... --node-bin ... --npm-bin ... --pnpm-bin ... --yarn-bin ...`
  - Result: passed on GNU libc Linux `x86_64` and `aarch64`; patch schema was `envlock.patch.v1`, symlink layout was stable, and `state.v2.json` was written correctly.

- Multi-Node hostile-PATH routing:
  - Installed `Node 18 / 22 / 24` side-by-side
  - Forced default `PATH` to prefer the wrong Node
  - Ran `envlock plugin node apply --node-bin <selected-node> --npm-bin ... --pnpm-bin ... --yarn-bin ...`
  - Verified with:
    - `node -p process.version`
    - `pnpm exec node -p process.version`
    - `yarn node -p process.version`
  - Result: `node`, `pnpm exec node`, and `yarn node` all resolved to the explicitly selected Node despite the hostile default PATH ordering.

- Mixed manager and global-install flows:
  - `npm install -g cowsay`
  - `pnpm add -g zx`
  - `pnpm dlx cowsay@1.6.0`
  - `yarn global add http-server`
  - Result: global binaries were installed under envlock-managed bin directories and remained runnable from those paths.

- Corepack and `packageManager` projects:
  - `corepack enable`
  - `corepack prepare pnpm@10.30.3 --activate`
  - Project with `"packageManager": "pnpm@10.30.3"`
  - Ran inside the project:
    - `envlock plugin node apply --node-bin <selected-node> ...`
    - `pnpm --version`
    - `pnpm store path`
    - `pnpm exec node -p process.version`
  - Result: apply succeeded inside the project, runtime `pnpm` version matched the project's `packageManager` declaration, `pnpm store path` followed that runtime version, and Node still resolved to the selected `--node-bin`.

- Yarn project/corepack checks:
  - Project with Yarn managed through corepack and project metadata
  - Ran:
    - `yarn --version`
    - `yarn add left-pad`
    - `yarn node -p process.version`
    - `yarn config get cacheFolder`
    - `yarn config get globalFolder`
  - Result: runtime Yarn version matched the project metadata/corepack selection, Node still resolved through envlock, and cache/global directories stayed under plugin-managed paths.

- Failure and recovery drills:
  - Simulated stale lock and interrupted apply, then re-ran `envlock plugin node apply`
  - Forced missing binaries with `--node-bin /missing/...`
  - Forced read-only state dirs and unwritable log homes
  - Result: stale-lock and interrupted-apply cases recovered on retry; permission failures stayed explicit; stderr included `See log:` when a per-invocation log file was available.

- Local non-invasive dogfooding rehearsal on macOS:
  - Created a fresh temp root with isolated `HOME`, `ENVLOCK_HOME`, `ENVLOCK_LOG_HOME`, project dirs, state dirs, and side-by-side Node installs
  - Installed published beta via `sh scripts/install.sh --version v0.4.6-beta.10`
  - Re-ran the same `plugin node init/apply`, install, global-install, packageManager, and recovery combinations inside the temp root
  - Result: the same flows passed in a fully isolated macOS temp environment without touching the live local setup; this was used as a pre-dogfooding confidence check before replacing `fnm` with envlock.

### Release operator view

- Confidence basis: published-artifact install validation, hostile multi-Node routing, manager/global-install coverage, package-manager-aware corepack projects, and local fully isolated rehearsal all passed without new blockers.
- Known non-blockers: `envlock plugin node init` is still a required first step, `install.sh` does not automatically modify `PATH`, and some upstream tool notices/deprecation warnings can still appear.
- Out of scope for this release: musl/Alpine, long-running soak behavior, and future corepack implementation changes outside envlock's control.
