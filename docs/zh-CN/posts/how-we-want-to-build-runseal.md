我们想如何 build runseal，其实完全由一条宪法决定：runseal 必须是 `agent-native` 的。

这句话很容易说，也很容易被说空，所以最好先把它讲具体。

## 唯一宪法

`agent-native` 不是“面向 prompt 优化”，也不是“名字听起来适合 AI 工具”。它真正的意思是：系统必须把足够多的真实信息，用足够低的获取成本暴露出来，让一个足够强的 agent 在不靠猜的前提下使用它、理解它、排查它。

## 什么叫诊断友好

所以真正的标准，不只是 `help` 写得好不好，而是 `help + stdout + stderr` 这三层加在一起，能不能把 agent 指向正确的排查方向，并尽可能缩短排查路径。

如果一个顶级 agent 看完 `help`、正常输出和错误输出之后，仍然无法判断问题出在 runseal 这一层的哪里，那这个设计就已经失败了。

## 我们的底线

这也是为什么 `env + symlink` 必须被视为 runseal 不能后退的地板。

`env` 让上下文显式存在，`symlink` 让入口显式存在。环境管理里最容易被藏起来的两件事实——“当前世界长什么样”和“命令最终落到哪里”——都必须重新变得可见。

只要这两件事是可见的，剩下很多脏边界就都能继续被理解：cache 放在哪、global 状态落在哪、版本根目录在哪里、helper 管了哪些路径、PATH 是怎么拼出来的、失败边界在哪一层。

一旦这两件事又被编排层和魔法重新吞回去，系统就会再次变得不值得信任。

所以 `env + symlink` 不是一种实现偏好，而是 runseal 的最低地板。

## 为什么核心必须足够小

再往上，架构自然也就跟着定了下来。

核心必须永远小。

最好永远不要超过一万行。

原因不是“行数越少越高贵”，而是核心天然会承受抽象压力。一旦太多生态特定逻辑向内渗透，核心就会慢慢长满历史兼容、边角协议、偶然状态和难以审查的特殊分支。工具也许还能继续工作，但它已经不再清楚，不再容易被理解。

我们宁可强迫自己把核心压窄。

核心应该负责的是定位、组合、执行，而不是替每个生态做一层越来越厚的统一管理。它不应该成为一个万能 runtime manager，也不应该变成一台隐藏状态机，只是恰好暴露出一个 CLI。

## 脏活留给 helper

这就是 helper 存在的原因。

helper 的价值，不是优雅，而是隔离复杂性。

以 `:node` 为例，真正麻烦的从来不是“切一个 Node 版本”，而是先建立一个自举出来的 Node + npm baseline，再处理围绕它出现的一整圈脏活：`pnpm`、`yarn`、全局安装、cache、store、wrapper、home-like 目录、稳定 shim，以及它们如何落进一个仍然能被 runseal profile 消费的结构里。

这些复杂性是真实存在的，不应该被假装消失；但它们也不应该反过来决定核心的形状。

所以我们的原则很简单：所有脏活、生态特定、变化快的工作，都应该被推给 helper。

这不是因为 helper 本身有多美，而是因为它能为复杂性提供一个诚实的边界：复杂性可以存在，但不要假装自己是普遍抽象。

核心保持薄。helper 承担脏活。如果某个生态最终原生长成了 runseal-compatible 的样子，我们应该删除 helper 的复杂性，而不是把它当作传统继续供起来。

## 文档即产品

同一条宪法，也决定了我们如何建设文档。

我们会主动把核心信息暴露到 agent 获取成本最低的位置。这也是为什么关键入口、契约信息、导航结构会被放进 metadata、help、stdout/stderr、以及稳定的 docs 结构里。

这不是文档装饰，而是产品表面的一部分。

如果获取信息的成本太高，agent 就会回退到猜。一旦系统必须依赖猜，它就已经不再 agent-native。

所以当我们说，希望 runseal 最终能像 `gh`、`aws`、`kubectl` 一样，不只是说命令名要好看，而是说它的命令表面必须清楚、上下文模型必须显式、诊断路径必须短。

## 我们想要的 runseal

这就是我们想要的 build 方式。

小核心，显式上下文，稳定入口，脏活外推，命令表面能帮助 agent 快速诊断，文档主动降低获取成本。

除此之外，其它都可以往后排。
