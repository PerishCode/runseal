我们想做 runseal，不是因为这个世界缺少环境管理工具。

恰恰相反，这个世界已经有太多环境管理工具了。对 Node 有 `nvm`、`fnm`、`volta`，对 Python 有 `virtualenv`、`venv`、`pyenv`，对更重的隔离场景还有容器、镜像、dev container、Nix 一类方案。它们都解决过真实问题，也都在不同阶段帮助过一代开发者。

## 新的约束

问题不在于它们毫无价值。问题在于，Agentic 时代对“环境管理”提出了一个比过去苛刻得多的要求：信息必须可见，副作用必须可控，上下文必须可组合。

过去，很多工具通过“隐藏复杂性”来换取“简化交互”。这对人类用户是成立的。我们可以接受一点 shell 魔法，一点 PATH 注入，一点“当前终端大概已经切好了”的模糊状态。哪怕出问题，最多也只是花几分钟重新理解自己现在到底在哪个环境里。

但对 Agent 而言，被隐藏的信息不是细节，而是毒药。

一个 Agent 需要知道当前上下文是什么、入口落到哪里、哪些变量被改写、哪些路径被污染、哪些状态会在命令结束之后继续残留。如果这些关键信息被藏进 shell hook、隐式切换、全局状态或者“约定俗成的魔法”里，系统就会开始变得脆弱。执行也许还能成功一次，但它不再稳定，不再可组合，也不再可验证。

这就是为什么我们会越来越不满足于传统的多环境管理答案。不是因为它们不聪明，而是因为它们太依赖“相信它已经帮你处理好了”。

## 零副作用与完全可见

我们想要的不是更聪明的魔法。我们想要的是一种可选的零副作用方案，一种信息完全可见的方案。

这里，“零副作用”并不是说系统永远不写任何文件，而是说：环境切换本身不应该偷偷改写你的世界。一次上下文激活不应该默认污染后续 shell，会话结束之后不应该留下无法追踪的隐式状态。你应该始终可以决定什么时候进入它，什么时候离开它，什么时候只生成一个可见快照而不真正启用它。

而“信息完全可见”则意味着另一件事：我不仅知道自己在用哪个环境，我还知道它到底做了什么。

如果我写的是 `:test kubectl ...`，我就应该随时能够回答：

- `:test` 指向了哪组配置？
- 它改了哪些环境变量？
- 它把哪些可执行入口重定向到了哪里？
- 它的 cache、store、prefix、bin、global 状态落在什么地方？
- 它在退出后留下什么，或者不留下什么？

## 为什么不直接上容器

我们不是没有见过好的答案。容器化方案在很多方面其实做得非常漂亮。它们天然强调隔离、边界和显式上下文。在“我要把一整个系统装进一个确定盒子里”这件事上，Docker 和类似方案几乎是教科书级别的。

但问题在于，日常开发中的多环境管理，很多时候并不需要一整个容器。

很多时候，我们只是想：

- 给这一组命令一个可命名的上下文
- 把工具入口稳定地指向一个确定版本
- 把 cache / global / store / home 这些脏边界收拢起来
- 在不引入整套重型运行时的情况下，获得近似容器的可见性和可控性

## 为什么从 `env + symlink` 开始

也正是在这里，我们越来越清楚 runseal 想做的事情：

在 Agentic 时代，多环境管理的最小充分答案，不是更厚的编排层，而是 `env + symlink`。

`env` 解决的是上下文。

它显式声明 cache 放哪、home 用哪、prefix 指哪、PATH 怎么组成、store 应该写到哪里。它把原本藏在工具默认行为里的副作用，提升成可以被阅读、生成、审核和组合的配置。

`symlink` 解决的是入口。

它让一个命令到底落到哪个实现，变成一个稳定、可见、可切换的事实。你不需要再赌当前 shell 的 PATH 顺序，也不需要再猜测某个 runtime manager 此刻到底把“默认版本”切到了哪里。入口本身可以被固定、检查和替换。

两者合起来，才构成了一个真正适合 Agent 的环境答案：

- 上下文是显式的
- 入口是稳定的
- 副作用是收敛的
- 行为是可组合的

## 过渡期的 helper

runseal 想做的，不是再造一个“大而全的环境平台”。我们也不想做一个厚重的核心，把所有生态细节都重新抽象一遍。相反，我们越来越相信：核心应该足够薄，复杂性应该被明确地留在边缘。

这也是为什么 helper 会成为 runseal 的重要组成部分。

以 `:node` 为例，真正麻烦的从来不是“切一个 Node 版本”，而是围绕这个版本出现的一整圈脏活：`npm`、`pnpm`、`yarn`、全局安装、wrapper、cache、store、bin、global、node_modules，以及它们如何和 profile、env、symlink 协同工作。

这个具体案例我们单独写成了 `:node/` 系列的第一篇：[01 | Making npm i -g pnpm Sealable](/zh-CN/node/01-making-npm-i-g-pnpm-sealable)。

这些复杂性是真实存在的，不能被一句“自动处理”抹掉。但它们也不应该被塞进一个越来越厚的核心里。更合理的答案是：让核心负责定位、组合和执行；让 helper 去接住具体生态里的脏边界。

所以 runseal 的方向从来不是“更多魔法”，而是更少魔法；不是“更聪明地隐藏”，而是更诚实地显式化；不是“更厚的统一抽象”，而是“更薄的核心 + 更明确的 helper”。

如果说旧时代环境工具最常见的承诺是“别担心，我会替你搞定”，那么我们更想给出的承诺是另一种：

我不会替你隐藏上下文。

我会把上下文做成一个可以被看见、被引用、被切换、被 seal 的东西。

> <span class="post-note-title">一些碎碎念</span>
>
> Agentic 时代，多环境管理的答案是 `env + symlink`。
>
> 我很尊重 `nvm` 在我早期实践中带来的收益，但不可否认的是，它为了简化终端用户的交互体验，增加了太多魔法。
>
> 这些魔法在过去也许只是“有点绕”，但在 Agentic 时代会变成致命问题：它隐藏的信息就是毒药。
>
> `volta` 其实也存在相似的问题，就不赘述了。
>
> 根本上，我们需要的是一种可选的“零副作用” + “信息完全可见”的方案。
>
> 容器化方案其实把这件事做得很漂亮，但日常开发的多环境管理直接用 Docker，总有点用牛刀的感觉。
>
> 我相信有些人的偏好和我很像：`xxx :test kubectl ...`，而我随时可以知道 `:test` 做了什么。
>
> 我不想再把环境管理建立在“它大概帮我处理好了”的信任上。
>
> 我想要的是：上下文被命名、被显式展开、被稳定复用，而且随时可以撤销。
